# Meal Pool Architecture Рђћ Design Spec

**Date:** 2026-04-19  
**Status:** Approved

## Goal

Replace AI-generated free-text nutrition plans with a structured meal pool system. Claude selects meal IDs from a curated pool; all names, macros, ingredients, and reminders are resolved deterministically from the pool. Eliminates translation inconsistencies, hallucinated macros, and unpredictable output.

## Problem Being Solved

The current system asks Claude to invent meal names, macros, ingredients, and reminders in Hebrew or English on every generation. This causes:
- Mixed-language output (English words appearing in Hebrew plans)
- Hallucinated or inconsistent macro values
- Grocery items that don't match what was actually planned
- Unpredictable kosher compliance

## 1. Meal Pool Data Model

**File:** `web/data/meal_pool.json`  
**Format:** JSON array of meal objects  
**Target size:** ~140 meals (30 breakfasts, 40 lunches, 40 dinners, 30 snacks)  
**Cuisine bias:** Israeli / Mediterranean, international variety

### Meal object schema

```json
{
  "id": "dinner_042",
  "type": "breakfast|lunch|dinner|snack",
  "name_en": "Shakshuka",
  "name_he": "ОЕОДОЕОЋОДОћ",
  "kcal": 420,
  "macros": { "protein_g": 22, "carbs_g": 18, "fat_g": 28 },
  "cooking_time": "quick|medium|elaborate",
  "positive_tags": ["kosher", "vegetarian", "gluten-free"],
  "ingredient_flags": ["contains_eggs", "contains_dairy"],
  "ingredients": [
    { "item_en": "Eggs", "item_he": "ОЉОЎОдОЎОЮ", "qty": "4", "category": "protein" },
    { "item_en": "Canned tomatoes", "item_he": "ОбОњОЉОаОЎОЋОф ОъОЕОЋОъОеОЋОф", "qty": "400g", "category": "pantry" },
    { "item_en": "Feta cheese", "item_he": "ОњОЉОЎОаОћ ОцОўОћ", "qty": "80g", "category": "dairy_fats" }
  ],
  "prep_note_en": null,
  "prep_note_he": null
}
```

### Positive tags (exhaustive list)
`kosher`, `vegetarian`, `vegan`, `keto`, `gluten-free`, `dairy-free`

### Ingredient flags (exhaustive list)
`contains_meat`, `contains_dairy`, `contains_pork`, `contains_shellfish`,
`contains_gluten`, `contains_eggs`, `contains_nuts`, `contains_soy`, `contains_fish`

### Ingredient categories (for grocery grouping)
`protein`, `vegetables`, `dairy_fats`, `pantry`

### IDs
Sequential per type: `breakfast_001`Рђд`breakfast_030`, `lunch_001`Рђд`lunch_040`,
`dinner_001`Рђд`dinner_040`, `snack_001`Рђд`snack_030`.

### Initial pool generation
The initial `meal_pool.json` is generated by Claude in a single implementation step,
following the schema above, reviewed by the developer before committing. No meals are
committed to the pool without human review.

## 2. Meal Selector

**File:** `web/meal_selector.py`

### Filtering logic

Given a user profile, produce an eligible meal list:

1. Load `meal_pool.json` (cached in module-level variable after first load)
2. **Exclusion filtering** Рђћ remove any meal where an `ingredient_flag` matches a user restriction:
   - Allergy "dairy" Рєњ exclude `contains_dairy`
   - Allergy "gluten" Рєњ exclude `contains_gluten`
   - Allergy "nuts" Рєњ exclude `contains_nuts`
   - Allergy "eggs" Рєњ exclude `contains_eggs`
   - Allergy "soy" Рєњ exclude `contains_soy`
   - Allergy "shellfish" Рєњ exclude `contains_shellfish`
   - Allergy "fish" Рєњ exclude `contains_fish`
   - Kosher Рєњ exclude `contains_pork`, `contains_shellfish`; also exclude any meal that has both `contains_meat` AND `contains_dairy`
3. **Cooking time filtering** Рђћ if user chose "quick", exclude `elaborate`; if "elaborate", include all
4. **Merge personal suggestions** Рђћ load the user's `MealSuggestion` rows with `status="pending"`, parse `meal_json`, run same exclusion filter, append to eligible list

### Claude selection prompt

Send Claude a compact text list grouped by meal type, IDs and English names only:

```
Build a 7-day meal plan by selecting from the eligible meals below.

Rules:
- No protein source repeated more than twice in a row across any meal type
- Vary cooking methods across the week
- User goal: {goals}
- Meals per day: {meals_per_day}
- Week dates (SunРђЊSat): {dates}

Return ONLY valid JSON: {"days":[{"date":"YYYY-MM-DD","meals":[{"type":"...","meal_id":"..."},...]}]}
No explanation, no markdown.

breakfast: breakfast_001 Shakshuka, breakfast_002 Oatmeal with banana, ...
lunch: lunch_001 Grilled chicken salad, lunch_002 Hummus plate, ...
dinner: dinner_001 Baked salmon, dinner_002 Chicken stir-fry, ...
snack: snack_001 Apple with almond butter, ...
```

### ID resolution

After Claude returns meal IDs, `meal_selector.resolve(schedule, pool)` maps each
`meal_id` to the full meal object. Unknown IDs are logged and skipped (graceful
degradation Рђћ the plan renders with whatever was resolved).

## 3. Groceries (Deterministic)

After resolving the weekly schedule, aggregate all `ingredients` arrays:

1. Group by `item_en` (case-insensitive match)
2. For duplicate items: concatenate quantities if units differ, sum if same unit
3. Sort within each `category` group
4. Output: same structure the current UI consumes Рђћ `[{item, qty, category}]`

`item_he` is stored alongside `item_en` and the template renders whichever matches `lang`.

## 4. Reminders (Deterministic)

After resolving the weekly schedule:

1. Collect all meals with a non-null `prep_note_en` / `prep_note_he`
2. Attach to the day the meal falls on (reminder fires the evening before Рђћ subtract 1 day)
3. Keep up to 3 reminders, prioritising earlier in the week
4. Store in `plan_json` under `"reminders"` Рђћ same schema as today: `[{day, text}]`
5. `text` is `prep_note_he` if `lang == "he"`, else `prep_note_en`
6. `Notification` rows created from reminders Рђћ body stored in user's language at generation time

## 5. User Suggestion Flow

### New DB model: `MealSuggestion`

```python
class MealSuggestion(Base):
    __tablename__ = "meal_suggestions"
    id: str              # UUID PK
    user_id: str         # FK Рєњ users.id
    status: str          # "pending" | "approved" | "rejected"
    meal_json: str       # Full meal object JSON (same schema as pool)
    created_at: datetime
```

### Submission UI

"Suggest a meal" button on `/nutrition` page opens a modal with a single textarea.
User describes the meal in natural language (any language). On submit:

1. `POST /nutrition/suggest` with `{"description": "..."}` 
2. Route sends description to Claude with enrichment prompt (see below)
3. Route returns structured JSON to browser
4. Browser renders a pre-filled editable confirmation form
5. User reviews/corrects and confirms
6. `POST /nutrition/suggest/confirm` saves the `MealSuggestion` row

### Claude enrichment prompt

```
Given this meal description, return a single JSON object with exactly these fields:
name_en, name_he, type (breakfast|lunch|dinner|snack),
kcal (integer), macros ({protein_g, carbs_g, fat_g} integers),
cooking_time (quick|medium|elaborate),
positive_tags (array, subset of: kosher/vegetarian/vegan/keto/gluten-free/dairy-free),
ingredient_flags (array, subset of: contains_meat/contains_dairy/contains_pork/
  contains_shellfish/contains_gluten/contains_eggs/contains_nuts/contains_soy/contains_fish),
ingredients (array of {item_en, item_he, qty, category}),
prep_note_en (string or null), prep_note_he (string or null).

No explanation. Raw JSON only.
Meal description: "{user_input}"
```

### Visibility

A user's `pending` suggestions are merged into their eligible pool at plan-generation
time. They appear in their own plans immediately after submission. Other users never
see pending suggestions.

## 6. Graduation Script

**File:** `scripts/graduate_meals.py`

CLI tool run manually by the developer:

```
python scripts/graduate_meals.py
```

For each `pending` MealSuggestion (ordered by `created_at`):
- Print the full meal JSON in a readable format
- Prompt: `[a]pprove / [r]eject / [s]kip`
- Approve: assign next sequential ID, append to `meal_pool.json`, set DB status to `approved`
- Reject: set DB status to `rejected`
- Skip: leave as `pending` for next run

Approved meals become available to all users on next server restart (pool is module-cached).

## 8. What Changes in `nutrition_generator.py`

- `_build_days_prompt` Рєњ replaced by `meal_selector.build_selection_prompt(user, eligible_meals)`
- `_build_groceries_prompt` Рєњ removed; groceries computed by `meal_selector.aggregate_groceries(resolved_meals)`
- `generate_weekly_plan` Рєњ calls `meal_selector` instead of two AI calls; one AI call remains (selection)
- Reminders logic Рєњ `meal_selector.extract_reminders(resolved_meals, week_dates, lang)`

## 9. Out of Scope

- Admin UI for editing pool meals (manage via the graduation script)
- User rating/upvoting of meals
- Seasonal or time-of-day meal weighting
- Nutritional goal tracking across the week (daily macro totals)
